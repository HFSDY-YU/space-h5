# H5 手机端技术选型

本文档用于确定 `space-h5/` 手机端的第一版技术栈。目标是基于现有澳琴空间预约系统后端，开发一个现代、类型安全、移动体验好、后续可持续维护的 H5 应用。

## 1. 结论

推荐技术栈：

| 分类 | 选型 | 说明 |
|---|---|---|
| 构建工具 | Vite 最新稳定版 | 使用官方 Vue 脚手架生成 Vite 工程，开发快、构建快，适合独立 H5 SPA。 |
| 框架 | Vue 3 | 与现有后台 Vue 技术栈同源，但升级到 Vue 3，降低团队迁移成本。 |
| 语言 | TypeScript | 移动端会大量对接后端接口，类型能约束请求、响应、状态和表单。 |
| 组件写法 | Composition API + `<script setup>` | 新项目默认采用 Vue 3 推荐写法，减少样板代码。 |
| UI 组件 | Vant 4 | 面向移动 Web 的 Vue 3 组件库，适合预约、表单、列表、弹窗、日期选择等 H5 场景。 |
| 路由 | Vue Router | 官方路由；生产优先 `history`，部署无法做 fallback 时切换 `hash`。 |
| 客户端状态 | Pinia | 保存登录态、用户信息、权限、轻量 UI 状态。 |
| 服务端状态 | TanStack Query for Vue | 管理列表缓存、刷新、重试、加载态、错误态和变更后失效。 |
| 请求层 | Axios + 类型封装 | 对齐后端 RuoYi `AjaxResult` / `TableDataInfo` / Bearer Token / 业务错误码。 |
| 样式 | Vant 主题变量 + UnoCSS + CSS 变量 | 用组件库承载基础交互，用原子化工具和设计变量控制页面密度与移动端适配。 |
| 表单校验 | Vant Form + Zod | 简单交互用 Vant 表单规则，跨页面复用的业务 schema 用 Zod。 |
| 测试 | Vitest + Vue Test Utils + Playwright | 单测覆盖工具、请求封装、状态；Playwright 做移动视口关键流程验证。 |
| 包管理 | pnpm + Corepack | 锁定依赖、安装快、适合后续 monorepo 或多前端目录管理。 |

推荐第一版目录名：

```text
space-h5/
```

原因：命名短、ASCII、适合脚本和部署路径；与现有 `ruoyi-ui/` 后台管理端区分明确。

## 2. 为什么选择 Vue 3 而不是继续 Vue 2

现有后台 `ruoyi-ui/` 是 Vue 2 + Element UI，适合管理后台，但新建 H5 手机端不需要继续背负 Vue 2 生态限制。

选择 Vue 3 的原因：

- 官方已经推荐新项目使用 Vue 3，Vue 3 在体积、性能、可扩展性和 TypeScript 支持上更好。
- Vue 官方文档说明 Vue 自身使用 TypeScript 编写，并提供一等 TypeScript 支持。
- 后台已有 Vue 使用经验，团队可以复用响应式、组件、路由、状态管理等心智模型。
- 手机端不需要兼容 IE11，可以使用现代浏览器能力。

不选择 React 的原因：

- React 生态也先进，但会引入另一套组件、状态和团队协作心智。
- 当前后端、后台和 RuoYi 前端链路已经是 Vue 体系，新 H5 用 Vue 3 更利于统一维护。

不选择 uni-app 的原因：

- 当前需求明确是 H5 手机端，不是一次性发布多端小程序/App。
- 纯 H5 SPA 用 Vite + Vue 3 更轻，调试和构建链路更直接。

## 3. 版本与运行环境

建议：

- Node.js：`>=22.12.0`，最低不低于 `20.19.0`。
- pnpm：使用当前稳定大版本，建议通过 Corepack 管理。
- Vite：使用最新稳定版，不手写旧版本模板。
- Vue：使用 `create-vue` 当前生成的 Vue 3 版本。

当前本机记录：

- Node.js：`v22.14.0`。
- npm：`11.3.0`。
- pnpm：`11.6.0`，当前通过用户级 npm 全局安装到 `C:\Users\uu\AppData\Roaming\npm\pnpm.ps1`。

原因：

- Vite 官方当前要求 Node.js `20.19+` 或 `22.12+`。
- Vue 官方快速开始也要求安装 `^20.19.0 || >=22.12.0` 的 Node.js。
- Vite 当前支持版本中已经包含 `vite@8.0` 常规补丁线，后续应按官方迁移指南定期升级。

初始化建议命令：

## 4. 移动端 UI 与交互

### 4.1 UI 组件

选用 `Vant 4`。

适配本项目的原因：

- Vant 面向移动 Web 应用，常用组件覆盖按钮、表单、单元格、弹窗、日期时间选择、下拉刷新、列表加载、标签、步骤、空状态等。
- 空间预约 H5 的核心页面以表单、列表、日期、审核操作为主，Vant 组件匹配度高。
- Vant 4 对 Vue 3 友好，适合新工程。

### 4.2 页面体验原则

- 首页不做后台式大仪表盘，优先提供“我要预约”“我的预约”“占用查询”“待审核”四类移动入口。
- 列表页必须支持下拉刷新、滚动加载、空状态、错误重试和筛选收起。
- 表单页必须固定底部提交栏，并处理安全区 `env(safe-area-inset-bottom)`。
- 审核操作必须二次确认，驳回和取消驳回应要求填写原因。
- 占用查询优先用日期、楼栋、房间类型、时段过滤，不把后台复杂表格搬到手机端。

## 5. 状态与数据获取

### 5.1 Pinia 负责客户端状态

适合放入 Pinia：

- Token。
- 当前用户信息。
- 权限集合。
- 当前选中的首页角色或快捷入口。
- 轻量 UI 设置，如主题、最近筛选条件。

不适合放入 Pinia：

- 后端列表缓存。
- 预约详情缓存。
- 统计接口缓存。
- 正在提交的服务端 mutation 状态。

### 5.2 TanStack Query 负责服务端状态

适合使用 TanStack Query：

- 房间列表、公开房间列表。
- 预约时段列表。
- 我的预约列表。
- 预约详情。
- 待审核列表。
- 提交预约、取消预约、审核通过、审核驳回后的缓存失效。

这样做的收益：

- 避免手写大量 `loading/error/list/refresh` 状态。
- 列表和详情可复用缓存。
- mutation 成功后可以精确失效相关查询。
- 对移动弱网场景可以统一处理重试和错误提示。

## 6. 请求层约定

请求层必须对齐当前后端：

- 登录：`POST /login`。
- 验证码：`GET /captchaImage`。
- 用户信息：`GET /getInfo`。
- 预约申请：`POST /space/reservation`。
- 我的预约：`GET /space/reservation/my/list`。
- 公开房间：`GET /space/room/public/list`。
- 公开预约汇总：`GET /space/reservation/public/summary/list`。
- 公开预约明细：`GET /space/reservation/item/public/list`。

当前 H5 开发环境接入方式：

- `.env.development` 设置 `VITE_API_BASE_URL=/dev-api`。
- `vite.config.ts` 将 `/dev-api` 代理到 `http://localhost:8080`，并在转发前移除 `/dev-api` 前缀。
- `src/api/http.ts` 创建 Axios 实例，统一注入 `Authorization: Bearer <token>`，识别 RuoYi `AjaxResult`、`TableDataInfo`、`401`、`500` 和业务错误。
- `src/api/auth.ts` 封装验证码、登录、用户信息和登出。
- `src/api/space.ts` 封装房间、时段、预约场次、我的预约、审核待办、预约详情、预约新增、取消和审核操作。
- `src/services/spaceMapper.ts` 将后端房间、时段、预约场次和预约主表转换为移动端页面模型。

封装要求：

- 所有需要登录的请求自动添加 `Authorization: Bearer <token>`。
- 统一识别 `AjaxResult.code`，`200` 成功，`401` 跳登录，其他错误走移动端 Toast。
- 分页接口统一转换 `rows`、`total`、`pageNum`、`pageSize`。
- 上传、导出等非移动端高频能力暂不优先实现。
- 不在页面里直接拼接 Axios；页面只能调用 `src/api/**` 或 hooks。

## 7. 推荐工程结构

后续脚手架完成后建议组织为：

```text
space-h5/
├── README.md
├── 技术选型.md
├── package.json
├── index.html
├── vite.config.ts
├── tsconfig.json
├── src/
│   ├── api/
│   │   ├── http.ts
│   │   ├── auth.ts
│   │   ├── space.ts
│   │   └── storage.ts
│   ├── components/
│   ├── layouts/
│   ├── router/
│   ├── services/
│   ├── stores/
│   ├── styles/
│   ├── types/
│   ├── views/
│   └── main.ts
├── src/__tests__/
└── e2e/
```

目录边界：

- `api/`：只放请求函数、请求类型和响应类型。
- `services/`：放后端 DTO 到移动端页面模型的转换、日期格式化和状态映射等纯业务适配。
- `views/`：放路由页面，不沉淀跨页面业务逻辑。
- `stores/`：只放客户端持久状态和跨页面 UI 状态。
- `components/`：放纯展示或轻交互组件，不直接调用业务接口。
- `types/`：放后端领域对象、枚举和通用响应类型。

## 8. 第一阶段功能范围

优先实现：

1. 登录、验证码、登出、Token 续用和 401 回登录。
2. 首页快捷入口。
3. 公开房间和公开占用查询。
4. 普通预约申请。
5. 我的预约列表、预约详情、取消申请。
6. 待审核列表、审核通过、审核驳回。

当前已实现真实接口接入：

- 登录页加载后端验证码，提交 `/login`，登录成功后调用 `/getInfo` 并按角色/权限识别学生、老师、管理员。
- 首页使用公开房间、公开时段和预约场次接口生成占用矩阵，状态仅展示空闲、审核中、占用。
- 房间详情使用公开房间详情、公开场次和公开预约汇总接口展示当天占用与公示。
- 老师预约申请走 `POST /space/reservation`，我的预约、预约详情和取消申请走对应真实接口。
- 管理员审核待办走 `/space/reservation/pending/list`，通过和驳回走主预约审核接口。
- 房间轻管理当前接入真实房间列表展示，启用/停用仍保留到后续移动端权限与完整编辑请求体确认后实现。

暂缓实现：

- 后台资源管理 CRUD。
- 房间 Excel 导入导出。
- 系统用户、角色、菜单管理。
- 代码生成。
- 复杂统计后台视图。

原因：H5 手机端应服务移动高频路径，不应复制完整后台管理端。

## 9. 质量要求

- 所有接口类型必须显式声明，避免页面消费 `any`。
- 所有中文文件保持 UTF-8。
- 移动端页面必须验证 375px、390px、430px 宽度。
- 关键流程用 Playwright 移动设备模拟验证。
- 提交预约、取消、审核这类 mutation 必须防重复点击。
- 表单提交失败必须保留用户输入。
- 公开接口展示字段必须遵守后端隐私清理边界，不自行展示被清理的内部字段。

## 10. 官方资料依据

- Vite 官方文档：Vite 是面向下一代 Web 的构建工具，当前指南要求 Node.js `20.19+` 或 `22.12+`，并推荐通过脚手架创建项目。
- Vue 官方文档：Vue 3 是新项目推荐选择，Vue 对 TypeScript 提供一等支持，官方脚手架可创建基于 Vite 且支持 TypeScript 的项目。
- Vant 官方文档：Vant 4 是面向移动 Web 应用的 Vue UI 组件库。
- Pinia 官方文档：Pinia 是 Vue 的状态管理库，具备类型安全、模块化和可扩展能力。
- TanStack Query 官方文档：TanStack Query 用于管理服务端状态、缓存、刷新、重试和 mutation 工作流。
- UnoCSS 官方文档：UnoCSS 是即时按需的原子化 CSS 引擎，提供 Vite 集成。
- Vitest 官方文档：Vitest 是 Vite 原生测试框架，能复用 Vite 配置与插件。
- Playwright 官方文档：Playwright 支持移动设备视口、触摸和浏览器行为模拟，适合 H5 关键流程验证。
